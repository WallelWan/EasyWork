# EasyWork 功能问题记录

以下为对当前实现的静态审阅问题清单，覆盖功能边界、潜在 bug、以及跨模块耦合风险。

## 执行图与生命周期

- 连续 run() 仅重建 taskflow，节点内部缓冲与状态未显式清理，可能导致重入运行残留数据。
- open/close 仅在 Python 层阻止 kwargs，C++ 层未做并发保护，多线程调用存在竞态风险。

## 调度与方法分发

- Mux 控制信号仅支持 bool/int，其他类型会静默落入默认分支，未抛错。
- Mux 丢弃逻辑直接 pop 其他分支队列，控制信号快速切换时可能丢弃错配数据。
- SyncBarrier 缓冲模型独立于 BaseNode，复杂拓扑下可能出现行为不一致。

## 类型系统与转换

- Pipeline.validate() 仅允许类型严格相等，但运行时支持类型转换，导致“运行时可连、验证失败”的不一致。
- double -> int64/float 的转换被默认允许，存在精度丢失风险。

## 节点注册与参数解析

- 注册参数解析失败时静默使用默认值，用户传错类型不会报错，易导致隐蔽问题。

## Python 绑定与 Eager/Tracing

- tuple 解包仅在 pipeline 模式可用，eager 模式直接报错，行为需明确说明。
- NodeWrapper 动态方法与属性名冲突时可能遮蔽访问。

## AST 控制流与分支

- AST 允许嵌套 if，但 BranchCtx 禁止嵌套分支上下文，可能触发运行时异常。
- if 条件类型未在构图期校验，运行时仅支持 bool/int，其他类型可能静默走 false。
- MergeNode 仅取第一个非空包，缺少按时间戳或控制信号的合并策略，可能丢数据。

## 控制节点与示例节点

- IfNode 条件缺失时不报错，默认走 false 分支。
- NumberSource/PairEmitter 在停止条件后仍返回默认值 (0 或空字符串)，可能污染下游。

## 耦合风险汇总

- 类型验证与运行时转换机制不一致。
- AST 语义与 BranchCtx 执行约束不一致。
- 控制节点类型约束与构图期检查不一致。

## 修复状态

- 类型校验支持可转换类型（若扩展暴露转换器）。
- IfNode 条件与 Mux 控制类型在构图期强校验。
- 参数解析改为严格模式，错误类型直接抛错。
- 分支上下文支持嵌套 if。
